Introduction

There are a lot of programming languages out there offering the ability to create dynamic yet safe code. Maybe you already have a favorite one! A common characteristic of these languages is that they require extensive familiarization and their safety features may be harder to work with compared to unsafe alternatives. For example, memory safety typically requires modifications to pure algorithmic logic that worry about non-functional code characteristics, like deadlocks and race conditions.

Blombly is a language with small instruction set and algorithm-only semantics. This means that it is limited to a few operations that cover a broad range of scenarios, whereas non-algorithmic tasks like memory management and parallelization are automated under-the-hood. The main language is compiled into an intermediate respresentation and is interpreted by an accompanying BLOck aseMBLY Virtual Machine (blomblyVM). The latter parses block-based assembly instructions that ensure reliability through the absense of jump statements; flow constrol is performed through code blocks of asembly instructions that are easier to apply safety constraints on.

So, who is blombly for? It may be a nice tool if your aim is one of the following:

✅ Quickly learn a dynamic (duck-typed) scripting language.
✅ Write sequential code that runs in parallel.
✅ Ensure memory safety by design.

A short demo

Below is a small blombly program that showcases several base language features. First, there is a code block that creates a scale variable with a provided default, if it does not already exist (we will see momentarily how it might exist). Then, the code block creates an error message if a non-positive scale is provided. The code block declaration does not run any code, but only creates a variable holding the block for future use. It is made final so that it cannot be modified. Similarly, a code block that performs addition and multiplication with some scale is provided. This takes as input two positional arguments x,y. This block also uses the scaler: symbol to inline the namesake. Inlining constitutes of immediately calling the other block's code, as if it were pasted already.

Inlining is further enriched by enclosing it an a try statement (the brackets can be inferred by the language, but they are there for clarity). This statement is responsible for intercepting errors and inline return values, and then places the intercepted values in the variable signal. The catch catch mechanism is the equivalent of if, with the difference that it is entered if it encounters an error instead of boolean truth. For safety, unhandled error values create further errors if they are never handled.

The code block is called like a function with positional arguments. But there can also be statements that create more variables after a | separator. These statements look like positional variables separated by semicolons, but by being fully fleshed code they offer greater flexibility. For example, they can inline the same configuration code block across several methods. Any variables created after | are transferred to the called add block together with the positional arguments. Blocks can be called with one or no positional arguments too. They may be also scheduled to run in autonomous execution in threads. While running, they have access only to their positional arguments, transferred variables, and final variables. 

final scaler = {
    default scale = 1;
    if(scale<=0)
       error("Scale should be greater than zero, but another value was given: "+str(scale));
}

final add(x,y) = {
    signal = try{scaler:}
    catch(signal) {
        print("handled an error");
        return 0;
    }
    return (x + y)*scale;
}

result = add(1, 2 | scale=0);
print(result);
